# DEPLOYMENT — sysperf

**This is THE runbook. No other doc explains how to build or run this project.**

`sysperf` is a **Linux-only desktop GUI**. It reads `/proc`, `/sys`, NVML and DRM `fdinfo`, so it
**cannot compile or run on Windows** — the Windows Drive is source-of-truth storage only. Build
and run on Ubuntu (native or **WSL2**).

---

## Prereqs

- Ubuntu 22.04+ (or any distro with Qt 6.2+).
- Toolchain + libs:
  ```bash
  sudo apt update
  sudo apt install build-essential cmake qt6-base-dev libdrm-dev
  ```
  - `build-essential` — gcc/g++ (C++17), make.
  - `cmake` — 3.21+ (for presets; 3.16+ works without presets).
  - `qt6-base-dev` — Qt 6 Widgets (dynamic-link, LGPLv3).
  - `libdrm-dev` — listed for completeness; the fdinfo collector parses files and needs no libdrm
    *linkage*. NVML is `dlopen`'d at runtime (no `-dev` package required to build).
- No environment variables are required. `.env.example` documents the optional ones.

## How to launch

```bash
./scripts/start.sh
```

`start.sh` runs `preflight.sh` (validates the project shape + toolchain), then `version.sh`
(writes `release.json`), then configures + builds with CMake into `build/`, then launches
`build/sysperf`. That is the only launch path.

Manual build (equivalent):

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j
./build/sysperf
```

Debug build with sanitizers (ASan + UBSan, per the plan):

```bash
cmake --preset debug-asan       # or: -DSYSPERF_SANITIZE=ON -DCMAKE_BUILD_TYPE=Debug
cmake --build build-debug-asan -j
```

### Running from WSL2

A GUI needs a display. On Windows 11 with WSLg this works out of the box; the app opens as a
native window. If you have no display you can still build and run the **headless tests** (below).

### Debugging collectors against fixtures

Every collector takes its procfs/sysfs **root as a parameter**, so you can point the app (or a
test) at a captured tree instead of the live system:

```bash
./build/sysperf --proc-root /path/to/proc --sys-root /path/to/sys
```

## How to verify it's running

- A window opens with a left rail (CPU / Memory / Disk N / Network N / GPU N) and a graph + stat
  grid per entry. Graphs fill left-to-right over ~60 s at 1 Hz.
- The status bar shows the app's **own** PSS and CPU% (self-instrumentation). If it exceeds the
  soft budget (PSS > 50 MB or CPU > 1% of one core) the readout turns red and a warning is logged.
- Cross-check numbers against `htop` / `free -m` / `nvidia-smi` / `nvtop` if available.

### Tests

```bash
ctest --test-dir build --output-on-failure
```

Runs the fixture-based collector unit tests (CPU/mem/disk/net/fdinfo/self parse + delta math) and
the budget smoke test. All are headless — no display required.

## How to stop

Close the window, or `Ctrl+C` the terminal running `start.sh`. No daemon, no PID file. The worker
thread is asked to stop and joined on exit.

## Packaging (scaffolding present, run on Linux)

- **`.deb`** (primary): `dpkg-buildpackage -us -uc -b` from the project root using
  `packaging/debian/`. `Depends:` are generated by `dh_shlibdeps` (never hand-written). Build in a
  clean chroot/container against the *target* release's Qt6.
- **AppImage** (portable fallback): `packaging/appimage/build-appimage.sh`. Build on the *oldest*
  supported glibc/Ubuntu so it runs everywhere newer; bundles the Qt platform plugin via
  `linuxdeploy` + its Qt plugin.
- **Snap/Flatpak: rejected** — strict confinement blocks other processes' `/proc`/`fdinfo` and
  device nodes, which would silently zero the GPU and cross-process collectors.

## When it breaks

1. Read what `preflight.sh` printed — the missing tool or drift is named.
2. `qt6-base-dev` not found ⇒ CMake `find_package(Qt6)` fails: install it.
3. GPU panel missing ⇒ expected on machines without that vendor's driver; NVML/DRM degrade
   gracefully by design.
4. Build dir stale ⇒ `rm -rf build && ./scripts/start.sh`.

Never bypass preflight. If you "had to", that's a bug in preflight — fix preflight.
